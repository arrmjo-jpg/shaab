# Execution Proof Rules — عقد إلزاميّ (مصدره صاحب المشروع)

أيّ مخالفة لهذه القواعد ⇒ التقرير غير موثوق والمهمّة تُعاد من جديد. الصدق مقدَّم على إكمال التقرير.

## 1) ممنوع ادّعاء الحفظ/التذكّر بلا دليل
لا يُقال «حفظت القاعدة / ستُحمَّل كلّ جلسة / اعتمدتها دائمًا / لن أنساها» إلا إذا كانت محفوظة فعليًّا
في مستودع المشروع أو ملفّ داخل المشروع أو نظام ذاكرة **يمكن التحقّق منه** (مسار يُفتح ويُقرأ)،
مع تقديم الدليل (مسار + محتوى مطبوع). سلوكُ التحميل التلقائيّ للجلسات القادمة لا يُدَّعى — يُذكر
المسار فقط ويُترك التحقّق للمستخدم.

## 2) كلّ ادّعاء تعديل يُرفق به
- قائمة الملفّات المعدَّلة بمساراتها الكاملة.
- أرقام الأسطر المتغيّرة.
- سبب كلّ تعديل.
- **الدليل الحقيقيّ** = الملفّ قبل التعديل (مستخرج) + الملفّ بعده (مستخرج)، أو `git diff`،
  أو Patch **مولَّد آليًّا من أداة التعديل**. الـPatch **المُركَّب يدويًّا ليس دليلًا بحدّ ذاته**
  بل عرضٌ للتغيير — يُرفق اختياريًّا للقراءة فوق المستخرجَين، لا بديلًا عنهما.
- نتيجة الفحوصات المناسبة (php -l / tsc / eslint / اختبار) **بمخرجاتها الخام**.

## 3) ممنوع إعادة بناء «قبل» من الذاكرة أو الاستنتاج
بلا Git/Baseline/Patch سابق يُكتب حرفيًّا:
> لا أستطيع إثبات الحالة السابقة، ويمكنني فقط إثبات الحالة الحالية.
الالتقاط الصحيح لـ«قبل»: طباعة المقطع بأمر ظاهر المخرجات **قبل** التعديل في نفس الجلسة
(مثل `awk 'NR>=a&&NR<=b'` مرقَّمًا)، ثمّ «بعد» بنفس الأمر، والـPatch يُركَّب منهما.

## 4) ممنوع عبارات النجاح الإنشائيّة بلا دليل ملاصق
«تم إثبات ذلك / تم التحقق بشكل قاطع / التنفيذ مكتمل / كل شيء يعمل / E2E PASS / tsc=0» —
لا تُكتب إلا متبوعة مباشرة بالمخرجات الخام التي تثبتها.

## 5) دليل سجلّ الجلسة الداخليّ يُعلَن عن طبيعته
إذا كان الدليل من سجلّ الجلسة الداخليّ فقط يُكتب حرفيًّا:
> هذا دليل مبني على سجل الجلسة الداخلية وليس دليلًا مستقلًا يمكن للمستخدم التحقق منه.

## 6) الصيغة الحرفيّة الإلزاميّة لتقرير نهاية كلّ مهمّة (أيّ تقرير يخالفها = غير مكتمل ويُعاد)

```
# 1) الملفات المعدلة
| الملف | Before | After | Diff | الفحص |
| ----- | ------ | ----- | ---- | ----- |
(لكلّ ملفّ ✅ أو ❌ حسب الدليل المتوفّر)

# 2) سبب التعديل
لماذا تم إجراء هذا التعديل.

# 3) Before
مقتطف حقيقيّ مستخرج من الملفّ قبل التعديل + الأمر المستخدم لاستخراجه (قابل لإعادة التشغيل).

# 4) After
مقتطف حقيقيّ مستخرج من الملفّ بعد التعديل + نفس طريقة الاستخراج.

# 5) Diff / Patch
git diff إن وُجد Git؛ أو Patch مولَّد آليًّا إن وُجد؛ وإلا يُكتب حرفيًّا:
> هذا Patch أعددته من المقاطع المستخرجة، وليس دليلًا تاريخيًا على حالة الملف قبل التعديل.
ولا يجوز اعتباره دليلًا.

# 6) نتائج الفحوصات
php -l / tsc / eslint / tests بمخرجاتها الفعليّة — أو التصريح بعدم وجود فحص مناسب.

# 7) ما لم أستطع إثباته
إلزاميّ دائمًا ولا يُحذف؛ «لا يوجد.» إن أمكن إثبات كلّ شيء، وإلا التصريح الصريح بكلّ نقطة.
```

## 7) عند تعذّر الإثبات
التصريح بالعجز بدل الاختلاق أو الاستنتاج — دائمًا.

## 8) ادّعاء إنشاء/تعديل ملفّ = دليل يتحقّق منه المستخدم
أيّ ادّعاء بإنشاء ملفّ أو تعديله (مثل «أُضيفت القاعدة X») يُصحَب **بالمقتطف الفعليّ من الملفّ
بعد التعديل** أو طريقة تحقّق مستقلّة (أمر قابل لإعادة التشغيل يطبع من القرص بالمسار وأرقام
الأسطر) — **الوصف وحده أو الـPatch المُركَّب وحده لا يكفيان**.
وبلا git، عند عرض Patch مُركَّب يُكتب حرفيًّا:
> هذا Patch أعددته من المقاطع المستخرجة، وليس دليلًا تاريخيًا على حالة الملف قبل التعديل.
وأيّ عبارة مثل «تم التنفيذ / تم إثبات ذلك / تم التحقق بشكل قاطع / حفظت هذه القاعدة دائمًا»
بلا دليل مرفق ⇒ يجب التصريح بدلها بأنّها غير قابلة للإثبات.

## 9) بلا Git أو Patch مولَّد آليًّا: ممنوع أفعال الإثبات القاطع عن الماضي
لا تُستخدم كلمات «أثبت / أكد / تم التحقق بشكل قاطع» عن الحالة السابقة، ويُكتب بدلها حرفيًّا:
> يمكنني إثبات الحالة الحالية فقط، أما الحالة السابقة فلا أملك لها دليلًا مستقلًا.

## 10) وضع التشغيل الافتراضيّ: تنفيذ مباشر (يَجُبّ ما يخالفه أعلاه)
- لا تقارير طويلة، لا فلسفة/Discovery/تحليل سرديّ، لا «تم التحقق/سأقرأ/أحتاج إذنك»، لا تقييمات رقميّة.
- نفّذ المطلوب مباشرة. عند الحاجة لقرار: سؤال واحد واضح ثمّ انتظار الإجابة.
- عند الانتهاء يُرسَل فقط: قائمة الملفّات المعدَّلة + الخطأ باختصار إن وُجد + «تم التنفيذ».
- الصيغة السباعيّة (القسم 6) تُستخدم فقط عند طلب الإثبات صراحةً.
